1. Project Overview

In modern machine learning engineering, shipping a model is only the beginning. Production systems must continuously monitor data quality, detect drift, respond to failures, retrain models automatically, and notify stakeholders — all without manual intervention.

This project tasks you with building exactly such a system, end-to-end.

You will:

Ingest a changing dataset from a provided HTTP API endpoint that simulates real-world data evolution (schema changes, distribution shifts, intermittent unavailability).
Train a machine learning classifier and deploy it as a REST API on AWS.
Expose Prometheus-compatible metrics from your ML service.
Set up a full observability stack (Prometheus + Grafana + Alertmanager) using Docker Compose.
Configure real-time Slack alerts for critical system events.
Implement automated model retraining and redeployment when model quality degrades.
Automate your build, test, and deploy pipeline using GitHub Actions CI/CD.
Note

This project is intentionally scoped to require 8–9 hours of focused work. Plan your time across the sections below. It is strongly recommended to complete tasks in the order presented, as later sections depend on earlier ones.

3. Part 1 — Data Ingestion and Schema Monitoring

Estimated Time: 1.0 – 1.5 hours

3.1 Data Source

Rather than a fixed static dataset, this project uses a live HTTP endpoint that returns evolving batches of records.

This simulates a real-world data pipeline where upstream data can change at any time:

New features may appear
Old features may disappear
The service may become temporarily unavailable
Important

You are permitted to call only the following endpoint:

http://149.40.228.124:6500/records

Calling any other endpoints on this server is not permitted and may result in disqualification of the relevant task.

3.2 API Behavior
Success (HTTP 200)

Returns a JSON payload containing:

schema: a list of feature names currently in the dataset.
records: a batch of data rows matching the current schema.
Service Unavailable (HTTP 503)

The endpoint is intentionally configured to return 503 errors at intervals, simulating data source downtime.

Your system must:

Detect this
Log it
Increment the appropriate Prometheus counter
Fire a Slack alert
3.3 What Your Ingestion Script Must Do

Write a Python script or service (ingestion.py or similar) that:

1. Polls the API

Calls the /records endpoint at regular intervals (e.g., every 30–60 seconds or upon invocation).

2. Stores Data

Parses and stores each batch locally.

Examples:

CSV
Parquet
In-memory storage
3. Detects Schema Changes

Compare the current schema with the previous schema.

If:

A new feature appears:
Log it
Fire metric
Send Slack alert
A feature disappears:
Log it
Fire metric
Send Slack alert
4. Detects Distribution Drift

Compute simple statistics such as:

Mean
Standard Deviation

Compare against a baseline.

If drift exceeds a configurable threshold:

Fire drift metric
Send Slack alert
5. Handles 503 Responses

When API returns HTTP 503:

Log failure
Increment datalake_unavailable
Send Slack alert
6. Triggers Retraining

Trigger model retraining when:

Enough new data accumulates
Schema changes occur
Distribution drift is detected

ingestion/
├── ingestion.py

8.2 Required Alert Rules
Alert 1 — DataLakeUnavailable

Condition:

increase(datalake_unavailable[1m]) > 0

Message:

Data source returned 503.
Check API availability.
Alert 2 — FeatureAdded

Condition:

increase(feature_added[1m]) > 0

Message:

New feature detected in schema.
Retraining may be required.
Alert 3 — FeatureRemoved

Condition:

increase(feature_removed[1m]) > 0

Message:

Feature dropped from schema.
Verify pipeline compatibility.

Reply "continue" and I'll extract Pages 13–18 (remaining alerts, GitHub Actions CI/CD, tests, project structure, submission requirements, grading rubric, and bonus marks).

continue
Alert 4 — DistributionDrift

Condition:

distribution_drift_detected == 1

Message:

Data distribution drift detected.
Model may be stale.
Alert 5 — FeatureDriftDetected

Condition:

distribution_drift_detected > 0

Message:

Feature-level drift flagged.
Investigate upstream data.
Alert 6 — HighResponseLatency

Condition:

histogram_quantile(
  0.95,
  response_delay_seconds_bucket
) > 1.0

Message:

P95 response latency exceeded 1 second.
Alert 7 — LowModelAccuracy

Condition:

model_accuracy < 0.8

Message:

Model accuracy dropped below threshold.
Auto-retraining triggered.
8.3 Alertmanager Slack Configuration

Create:

alertmanager/alertmanager.yml

Route all alerts to Slack using a webhook URL.

The webhook URL must be stored as an environment variable:

SLACK_WEBHOOK_URL

Example:

receivers:
  - name: "slack-notifications"
    slack_configs:
      - api_url: "${SLACK_WEBHOOK_URL}"
        channel: "#mlops-alerts"
        send_resolved: true
        title: "[{{ .Status | toUpper }}] {{ .CommonLabels.alertname }}"
        text: "{{ range .Alerts }}{{ .Annotations.description }}{{ end }}"
